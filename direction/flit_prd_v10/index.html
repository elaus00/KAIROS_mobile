    Flit. — 제품 기획안 (PRD)¶

    Version: 10.1 작성일: 2026-02-12 플랫폼: Android (Kotlin, Jetpack Compose)

    0.A 2026-02-12 Addendum (우선 적용)¶

    본 PRD의 기존 Google Calendar/OAuth 중심 문맥은 아래 기준으로 대체 적용한다.

    1. 캘린더 연동
    2. 기존: Google OAuth + 서버 프록시 기반

    3. 현재(Android): CalendarProvider 기반 로컬 캘린더 연동

    4. 로그인

    5. 기존 표현: Google OAuth 일반 표현

    6. 현재 구현: Google Sign-In(ID Token) -> 서버 /auth/google -> JWT

    7. 로그아웃 정책

    8. 기존: 로컬 데이터 초기화 가능 전제

    9. 현재: 인증 토큰만 제거, 로컬 데이터 유지

    10. 일정 동기화

    11. 기존: 양방향 Google Calendar 동기화 중심
    12. 현재(Android): 로컬 캘린더 쓰기 중심 + /sync/* 기반 데이터 동기화로 확장

    1. 문제 정의¶

    1.1 핵심 문제 — 기록되지 않는 생각¶

    투두앱을 깔았다 지웠다를 반복하는 직장인, 메모가 쌓여 기록을 포기한 대학원생, 떠오른 생각을 30초 안에 외부화하지 않으면 잃어버리는 사람 — 이들이 공통으로 겪는 문제가 있다. 떠오른 생각이 기록되지 않고 사라진다는 것이다.

    사람들은 하루에도 수많은 생각을 한다. 그중 일부는 기록할 가치가 있지만, 대부분은 기록되지 않고 사라진다. 기록되지 않는 이유는 생각의 가치가 없어서가 아니라, 기록하고 관리하는 과정이 부담이기 때문이다. 실제로 스마트폰 사용자 중 전용 메모·캘린더 앱을 사용하는 비율은 12.7%에 불과하다 [1]. 도구가 없어서가 아니라, 기존 도구의 기록·정리 과정이 충분히 쉽지 않기 때문이다.

    1.2 왜 기록되지 않는가¶

    기록이 안 되는 근본 원인은 두 가지다.

    분류 판단의 피로

    생각이 떠오를 때마다 "이건 할 일인가, 일정인가, 아이디어인가?"를 판단해야 하고, 판단한 뒤에는 해당 유형에 맞는 도구를 선택해야 한다. 이 분류 판단 자체가 기록의 첫 번째 허들이다.

    특히 유형 간 경계가 모호한 경우가 많다. "금요일까지 보고서 제출"은 할 일인가, 일정인가? "다음 주 미팅에서 이 안건 제기"는 메모인가, 할 일인가? 이러한 모호함은 판단 에너지를 소모하게 만든다.

    여기에 더해, 분류의 기준은 사람마다 다르다. 같은 텍스트를 누군가는 할 일로, 누군가는 메모로, 또 누군가는 일정으로 인식한다. 즉, 분류 판단의 피로는 분류가 어려워서 생기는 것일 뿐 아니라, 정답이 하나가 아니기 때문에 생기는 것이기도 하다.

    정리 부채의 누적

    기록은 했지만 정리는 하지 못한 상태가 지속되면 두 가지 경로로 문제가 발생한다.

    경로 문제 결과
    기록 포기 정리 부담이 기록 행위 자체를 억제 생각이 유실됨
    정리 부채 기록은 하지만 정리가 누적 과제로 남음 기록의 가치를 회수하지 못함

    이미 정리되지 않은 메모가 수십 개 쌓여 있으면, 거기에 또 하나를 추가하는 것 자체가 부담이다. "어차피 또 안 정리할 텐데"라는 생각이 기록 행위를 막는다. 정리되지 않은 메모가 쌓이면서 생기는 만성적 피로감과 부채감이 결국 기록 자체를 포기하게 만드는 악순환이 만들어진다.

    이 악순환은 개별 사용자 수준을 넘어 시장 수준에서도 관찰된다. 한때 2.5억 등록 사용자를 보유했던 Evernote는 다운로드가 82% 감소하며 구조적 쇠퇴에 접어들었다 [2]. 정리되지 않는 메모 시스템이 결국 도구 자체의 포기로 이어진다는 것을 보여주는 사례다.

    1.3 기존 솔루션의 한계¶

    도구 유형 한계
    투두 앱 할 일을 추가하려면 앱 진입 → 추가 버튼 → 제목 입력 → 태그 선택 → 마감일 설정이라는 다단계 과정이 필요하다. 5,000만 사용자를 보유한 Todoist조차 자연어 입력으로 마찰을 줄였지만, "이것이 할 일인지 메모인지 일정인지"는 여전히 사용자가 판단해야 한다 [3]. 완료 처리도 귀찮아 방치된다.
    메모 앱 정리되지 않은 메모가 쌓이면서 기록 자체를 억제한다. Google Keep은 "디지털 정크 서랍"이라는 평가가 반복되며, 50개 라벨 제한의 플랫 구조는 노트가 많아질수록 정리가 불가능해진다 [4]. 피로감을 감수하고 기록을 계속해도 정리 부채는 누적된다.
    캘린더 앱 일정 정보는 대화(카톡, 문자 등)에서 발생하는 경우가 많지만, 캘린더에 반영하려면 별도로 앱을 열고 수동으로 입력해야 한다.
    도구 간 분산 여러 앱에 흩어진 정보가 어디에 있는지 기억하지 못한다. 사용자의 28%가 복수 앱 관리로 인한 피로를 겪고 있으며 [5], 북마크도 여러 곳에 저장해놓고 결국 보지 않는다.

    2. 제품 비전¶

    2.1 한 줄 요약¶

    "떠오른 순간, 바로 던지면 끝"

    Flit.은 떠오르는 생각을 즉시 기록하면, AI가 자동으로 일정·할 일·아이디어·노트로 분류하여 정리해주는 개인 생산성 앱이다.

    2.2 핵심 철학: Just Capture¶

    Flit.의 경쟁자는 다른 메모앱이 아니다. "나중에 적어야지" 하다가 잊어버리는 것, 그리고 "또 정리 안 할 텐데" 하면서 기록을 포기하는 것이다.

    Flit.은 기록과 정리 사이의 연결고리를 끊는다. 사용자는 기록만 하면 된다. 정리는 AI가 한다. Flit.의 목표는 더 나은 기록 도구가 아니라, 기록을 포기하지 않게 만드는 시스템이다. 정리 부채가 쌓이지 않으면 포기 신호가 발생하지 않고, 기록은 계속된다.

    원칙 설명
    Just Capture 앱 실행 → 즉시 입력. 어디에 적을지 고민할 필요 없이, 그냥 적으면 된다
    Auto Organize 분류·태깅·목적지 결정은 AI가 처리한다. 정리되지 않은 메모가 쌓이지 않는다
    Straight to Action 일정은 캘린더로, 할 일은 투두로 자동 연결된다. 기록에서 행동까지의 거리를 없앤다

    2.3 설계 판단 기준¶

    기능을 추가하거나 UI를 변경할 때 아래 질문에 하나라도 Yes이면 재고한다.

    기준 질문
    Friction 이것이 기록까지의 단계를 늘리는가?
    Choice 사용자에게 선택을 강요하는가?
    Burden 정리 부담을 사용자에게 전가하는가?
    Flow 핵심 루프(기록 → 정리 → 실행)를 방해하는가?
    Necessity 이것 없이도 앱이 동작하는가?
    Quality AI 분류 품질을 비용 절감이나 편의를 위해 타협하는가?

    설계 판단 기준의 구체적 적용 방법과 사례는 철학 및 원칙 가이드에서 다룬다.

    3. 타겟 사용자¶

    페르소나 A: 민수 (28, 직장인)¶

    투두앱 3개를 깔았다 지웠다를 반복. 카톡 약속을 캘린더에 안 옮겨서 더블부킹 경험 있음.

    항목 내용
    타겟 유형 투두앱을 방치하게 된 사람 / 대화 속 일정을 캘린더에 안 옮기는 사람
    핵심 Pain Point 기록 도구 간 전환 비용, 일정 누락
    앱 진입 목적 카톡에서 약속 텍스트를 공유 → 일정 자동 등록 / 떠오른 할 일을 빠르게 던져놓기
    주요 사용 기능 캡처, 공유 인텐트, 일정 관리 (로컬 캘린더 연동), 할 일 관리, 위젯

    페르소나 B: 서연 (24, 대학원생)¶

    연구 아이디어가 떠오르면 바로 적어야 하는데, 메모앱 열 때마다 정리 안 된 노트가 보여서 스트레스. 정리는 주말에 몰아서 하려고 하지만 실제로는 안 함.

    항목 내용
    타겟 유형 정리 안 된 메모가 쌓여 기록 자체가 부담인 사람 / 정리 부채가 누적된 사람
    핵심 Pain Point 정리 부채 누적, 기록 가치 미회수
    앱 진입 목적 아이디어가 떠오르면 분류 고민 없이 바로 기록 / 나중에 노트를 검색해서 활용
    주요 사용 기능 캡처, AI 자동 분류, 노트, 검색

    페르소나 C: 준혁 (31, 프리랜서 개발자)¶

    ADHD 진단. 하나의 작업에 과몰입(hyperfocus)하다가 갑자기 다른 생각이 끼어들면 원래 하던 맥락을 통째로 잃어버린다. 머릿속에 떠오른 것을 즉시 외부화하지 않으면 30초 안에 사라지는데, 외부화하려고 다른 앱을 여는 순간 "여기에 적을까, 저기에 적을까"를 고민하는 사이에 원래 생각을 잊는다.

    투두앱을 쓸 때 가장 큰 문제는 시간 인지의 왜곡(time blindness)이다. 마감일을 설정해도 "아직 여유 있겠지"라는 감각이 실제 남은 시간과 심하게 괴리되어, 마감 직전까지 착수하지 못한다.

    기록 도구가 여러 개 있으면 더 힘들다. 어떤 건 Google Keep에, 어떤 건 메모에, 어떤 건 노션에 적어놓고 나중에 어디에 적었는지 기억하지 못한다. "하나의 진입점"이 존재하지 않는 것 자체가 기록 행위의 마찰이 된다.

    항목 내용
    타겟 유형 주의력 관리에 어려움을 느끼는 사람 / 작업 기억 용량이 제한적이어서 즉시 외부화가 필요한 사람
    핵심 Pain Point (1) 생각이 떠오른 즉시 외부화하지 않으면 유실됨 (작업 기억 한계), (2) "어느 앱에 적지?"를 고민하는 순간 과제 전환 비용이 발생하여 원래 생각을 잃음, (3) 마감일의 자기 관리가 어려움 (실행 기능 결함, 시간 인지 왜곡)
    앱 진입 목적 생각이 떠오르는 즉시 한 곳에 던져놓기 / AI가 마감일을 대신 판단해주길 기대
    주요 사용 기능 캡처 (입력창 즉시 접근), 위젯 (입력/할 일), AI 자동 분류, AI 마감일 제안

    Secondary¶

    여러 생산성 도구를 사용하며 정보가 흩어져 있고, 이를 하나의 입구로 통합하고 싶은 사람.

    4. 기능 요구사항¶

    4.1 캡처 (Capture)¶

    사용자가 떠오른 생각을 최소한의 단계로 기록하는 기능. 앱 진입 시 입력창이 즉시 접근 가능한 위치에 고정되며, 전송 즉시 로컬 저장 및 완료 피드백이 제공된다. 사용자에게 유형 선택을 요구하지 않는다.

    입력 방식:

    방식 설명 Phase
    텍스트 직접 입력 앱 내 입력창에서 직접 작성 1
    이미지 첨부 갤러리 또는 카메라에서 이미지를 선택하여 텍스트와 함께 전송 2a

    세부 기능:

    이름 설명 Phase
    즉시 입력 앱 진입 시 별도 네비게이션 없이 입력창에 접근 1
    로컬 우선 저장 전송 즉시 로컬 DB에 저장, 네트워크 상태와 무관 1
    완료 피드백 저장 완료 시 시각적 피드백 (앱 내 알림) 1
    임시 저장 입력 중 앱 이탈 시 텍스트 자동 저장, 재진입 시 복원 1
    삭제 실행 취소 삭제 후 Snackbar로 "실행 취소" 옵션 제공 (3초), 이후 휴지통으로 이동 1
    휴지통 삭제된 항목을 30일간 보관, 복원 가능. 30일 경과 시 영구 삭제 2a
    이미지 첨부 갤러리 또는 카메라에서 이미지 선택하여 캡처에 포함 2a
    멀티 인텐트 분할 하나의 캡처에 여러 의도가 포함된 경우 AI가 자동 분할하여 별도 캡처 생성 (parent_capture_id 참조) 2b
    외부 공유 캡처 내용을 외부 앱으로 공유 (Intent.ACTION_SEND) 3b

    4.2 AI 자동 분류¶

    캡처된 내용을 AI가 분석하여 유형과 목적지를 자동으로 결정하는 기능. 사용자 개입 없이 백그라운드에서 처리된다.

    분류 유형:

    내부적으로 3개 최상위 유형(SCHEDULE, TODO, NOTES)과 임시 상태(TEMP)를 사용한다. 사용자에게는 4개 유형(일정, 할 일, 노트, 아이디어)으로 표시되며, 내부 구조와 표시 구조는 분리된다.

    내부 유형 (classified_type) 서브 유형 (note_sub_type) 사용자 표시 목적지
    SCHEDULE — 일정 캘린더 탭 → 로컬 캘린더(CalendarProvider) 동기화
    TODO — 할 일 캘린더 탭 할 일 섹션
    NOTES IDEA 아이디어 노트 / Ideas 폴더
    NOTES INBOX 노트 노트 / Inbox 폴더
    NOTES BOOKMARK (노트) 장기 로드맵 (북마크 기능 확장 시 활성화)
    TEMP — — 미분류 임시 상태 (재분류 대기)

    TEMP는 미분류 상태를 나타내는 임시 상태이며, 네트워크 미연결, AI 분류 실패, 텍스트 없이 이미지만 첨부된 경우 TEMP로 저장된다. AI가 주기적으로 재분류를 시도하여 적절한 유형으로 이동시킨다. 주기적 재분류는 WorkManager 기반으로 동작하며, 네트워크 복구 시에도 자동으로 TEMP 항목의 분류를 재시도한다.

    AI 분류 출력:

    AI는 분류 시 유형 판별과 함께 다음 항목을 생성한다.

    출력 설명 비고
    유형 SCHEDULE / TODO / NOTES 필수
    서브 유형 NOTES인 경우: INBOX / IDEA / BOOKMARK NOTES일 때 필수
    신뢰도 High / Medium / Low 일정 자동 추가 안전장치에 활용
    AI 제목 원문 기반 요약 제목 (최대 30자) 리스트 가독성 확보
    태그 내용 기반 자동 태그 검색·필터링에 활용
    핵심 엔티티 사람, 장소, 날짜, 금액 등 검색 키워드로 활용

    AI 제목은 캡처 원문이 길거나 대화체인 경우 리스트에서의 스캔 가능성을 위해 생성된다. 캡처 원문이 짧고 명확한 경우(예: "우유 사기")에는 원문을 그대로 제목으로 사용한다. AI 제목이 생성된 경우에도 원본 텍스트는 항상 보존되며, 상세 보기에서 확인할 수 있다.

    비기록성 입력 처리:

    캡처는 "저장"의 영역이며, AI 대화 기능으로 확장하지 않는다. 질문형이나 대화형 입력("오늘 날씨 뭐야?", "좋은 카페 추천해줘")은 "확인할 것" 성격의 할 일로 분류하여 행동/기록으로 환원한다.

    입력 예시 분류 결과 비고
    "오늘 날씨 뭐야?" 할 일 (확인) / 마감: 오늘 질문을 행동 항목으로 전환
    "좋은 카페 추천해줘" 할 일 (알아볼 것) 조사 필요 항목으로 전환
    "내일 발표 준비해야지" 할 일 / 마감: 내일 일반적 할 일 분류

    분류 결과 확인:

    분류 결과는 AI 분류 현황 화면에서 일괄 확인할 수 있다 (4.3 참조). 사용자는 분류를 사후에 수정할 수 있으며, 수정 시 드롭다운에 4개 옵션(일정 / 할 일 / 노트 / 아이디어)이 표시된다.

    분류 수정 시 파생 객체 처리:

    사용자가 분류를 변경하면, 이전 분류에 의해 생성된 파생 객체(캘린더 이벤트, 투두 등)는 자동으로 삭제 또는 전환된다. 분류 수정 후 파생 객체가 불일치 상태로 남아 사용자가 직접 정리해야 하는 상황을 만들지 않는다.

    변경 방향 파생 객체 처리
    일정 → 할 일 캘린더 이벤트 삭제 + 투두 생성
    일정 → 노트/아이디어 캘린더 이벤트 삭제 + 노트 생성
    할 일 → 일정 투두 삭제 + 캘린더 이벤트 생성 (신뢰도 기반 처리 적용)
    할 일 → 노트/아이디어 투두 삭제 + 노트 생성
    노트/아이디어 → 일정 노트 삭제 + 캘린더 이벤트 생성 (신뢰도 기반 처리 적용)
    노트/아이디어 → 할 일 노트 삭제 + 투두 생성
    노트 ↔ 아이디어 note_sub_type만 변경 (INBOX ↔ IDEA), folder_id 업데이트

    파생 객체 처리의 세부 동작(삭제 확인 여부, 외부 캘린더 이벤트 삭제 타이밍 등)은 기능 명세서에서 정의한다.

    분류 기준 개인화:

    AI 분류는 범용적 기준으로 동작하지만, 분류의 기준은 사람마다 다르다. 동일한 텍스트를 어떤 사용자는 할 일로, 다른 사용자는 메모로 인식할 수 있다. 이 문제를 완화하기 위해 다음 수단을 제공한다.

    수단 설명 Phase
    사후 수정 분류 결과를 사용자가 직접 변경 1
    분류 수정 로깅 사용자의 분류 수정 이력을 기록 2a
    수정 이력 기반 학습 사용자의 분류 수정 패턴을 반영하여 분류 정확도 향상 (2b 초안, 3b 고도화) 2b~3b
    분류 프리셋 사전 정의된 분류 성향 프로필 제공 (예: "할 일 중심", "메모 중심") 3a (구독)
    사용자 지시 (Instruction) 사용자가 자연어로 분류 규칙을 직접 설정 3a (구독)

    분류 정확도가 사용자의 기대와 부합하는지 여부는 핵심 리스크이며, 유저 테스트를 통해 지속적으로 검증한다 (8. 리스크 & 미결 사항 참조).

    세부 기능:

    이름 설명 Phase
    자동 유형 판별 캡처 내용을 분석하여 SCHEDULE / TODO / NOTES 중 하나로 분류. NOTES의 경우 서브 유형(IDEA / INBOX / BOOKMARK) 추가 판별 1
    AI 제목 생성 캡처 원문 기반 요약 제목 생성. 원본 텍스트는 항상 보존 1
    자동 태깅 내용에 기반한 태그 자동 부여 (사용자 수정 가능) 1
    핵심 엔티티 추출 사람, 장소, 날짜, 금액 등 핵심 개체 추출 1
    AI 분류 현황 분류 완료된 항목을 전용 화면에서 일괄 확인/수정 (알림 벨 → 바텀시트) 1
    사후 수정 사용자가 분류 결과를 직접 변경 (4개 옵션 드롭다운) 1
    파생 객체 자동 처리 분류 변경 시 이전 파생 객체 자동 삭제/전환 1
    비기록성 입력 처리 질문형/대화형 입력을 행동 항목(할 일)으로 환원 1
    TEMP 폴백 미분류/오프라인 캡처는 TEMP 상태로 저장 1
    TEMP 주기적 재분류 TEMP 잔류 항목에 대해 AI가 주기적으로 재분류 시도 1
    분류 수정 로깅 사용자의 분류 수정 이력을 기록 2a
    멀티 인텐트 분할 여러 의도가 포함된 캡처를 AI가 자동 분할 2b
    수정 이력 기반 학습 수정 패턴을 반영하여 분류 정확도 향상 (2b 초안, 3b 고도화) 2b~3b
    분류 프리셋 사전 정의된 분류 성향 프로필 선택 3a (구독)

    5. 비기능적 요구사항¶

    5.1 데이터 원칙¶

    모든 콘텐츠 데이터(캡처, 노트, 일정, 할 일)는 로컬 우선(local-first)으로 저장한다. 캡처는 네트워크 상태와 무관하게 항상 즉시 성공해야 한다. 서버는 AI 처리(분류, 요약)와 계정·과금 데이터만 담당한다.

    5.2 오프라인 지원¶

    네트워크가 없는 상태에서도 캡처, 조회, 삭제가 가능해야 한다. AI 분류가 필요한 캡처는 TEMP 상태로 저장되며, 네트워크 복구 시 자동으로 분류 처리된다. 오프라인 상태에서의 사용자 경험 저하를 최소화하되, 오프라인에서 불가능한 기능(AI 분류, 서버 동기화)은 명확히 표시한다.

    5.8 TEMP 상태 및 Inbox 설계 원칙¶

    TEMP는 AI 미분류 항목의 임시 상태이며, 노트 탭 내 Inbox 폴더에 표시된다. TEMP 상태 자체가 "처리해야 할 대기열"로 인식되면 정리 부채의 원인이 되므로, 다음 원칙을 적용한다.

    원칙 설명
    비전면 배치 Inbox는 메인 네비게이션에 전면 노출하지 않는다. 노트 섹션 내부에 위치한다
    비침습적 알림 잔류 항목이 있을 때 "N개 항목이 분류 대기 중" 수준의 부드러운 nudging만 제공한다. 강제적 처리를 유도하지 않는다
    AI 자동 해소 AI 주기적 재분류가 TEMP 항목을 자동으로 분류하는 것이 기본 동작이다. 사용자가 Inbox를 직접 관리해야 하는 빈도를 최소화한다
    TEMP ≠ 분류 유형 TEMP는 사용자에게 별도 분류 칩으로 표시하지 않는다. 분류 대기 상태임을 나타내는 내부 상태이다

    5.9 분류 품질 관측¶

    AI 분류 품질은 서비스 신뢰의 핵심 요소이며, 출시 시점부터 관측 인프라를 구축한다. 정답 데이터 없이도 프록시 지표로 분류 품질을 측정할 수 있다.

    최소 관측 지표 (Phase 2a부터 수집):

    지표 측정 방법 의미
    분류 수정률 수정 건수 / 전체 분류 건수 AI 분류 품질의 핵심 대리 지표
    수정까지 걸린 시간 분류 완료 → 수정 실행 간 시간 오분류 인지 속도
    일정 제안 승인율 승인 / (승인 + 거부 + 무시) 캘린더 안전장치 정합성
    일정 자동 추가 후 삭제 비율 자동 추가 후 사용자 삭제 건수 / 자동 추가 건수 High confidence 임계값 적정성
    TEMP 잔류 시간 TEMP 진입 → 재분류/수동 이동까지 시간 미분류 처리 효율

    확장 관측 지표 (Phase 2b부터 수집):

    지표 측정 방법 의미
    캡처 후 재열람률 캡처 후 해당 항목 재조회 비율 기록 가치 회수 시그널
    투두 완료율 완료 / 생성 행동 전환 시그널
    검색 성공률 검색 후 항목 클릭 비율 회수 인프라 품질

    이 지표들은 리스크 #1(AI 분류 정확도)의 판단 근거이자, 수정 이력 기반 학습(Phase 2b 초안)의 효과 측정 수단이다.

    8. 리스크 & 미결 사항¶

    # 항목 설명 상태
    1 AI 분류 정확도 및 개인차 분류 기준은 사람마다 다르며, AI의 범용 분류가 모든 사용자에게 부합하지 않을 수 있음. 출시 후 수정률을 추적하며, 수정률 임계치 초과 시 개인화 수단 도입 일정을 앞당김. 유저 테스트 문서 참조 핵심 리스크
    2 일정 자동 추가 오분류 신뢰도(Confidence) 기반 조건부 처리로 대응 (4.5 참조). 기본값은 "제안" 모드. 신뢰도 임계값의 적정 수준은 유저 테스트를 통해 검증 예정 대응 중
    3 캡처 입력의 AI 처리 범위 캡처는 "저장"의 영역이며, AI 대화 기능으로 확장하지 않는다. 질문형/대화형 입력은 "확인할 것" 성격의 할 일로 분류하여 행동/기록으로 환원한다 (4.2 비기록성 입력 처리 참조) 방침 확정
    4 외부 서비스 동기화 확장 Obsidian 등 외부 도구 연동 가능성. 추후 동기화 아키텍처 설계 시 별도 논의 미결
    5 데이터 동기화 및 마이그레이션 크로스 디바이스 동기화, 외부 도구 데이터 임포트, 웹/PC 확장 시 콘텐츠의 서버 저장 및 동기화 설계 필요 미결
    6 성공 지표 세부 운영 핵심 가설(마찰 감소, 정리 부채 해결)과 지표 프레임은 설정됨. 세부 수치 기준은 출시 후 운영 데이터를 바탕으로 보완 미결
    7 로컬 캘린더 연동 상태 일관성 권한 변경/대상 캘린더 변경/수동 삭제 시 앱 상태와 캘린더 상태 간 불일치 가능성. 재동기화 정책 고도화 필요 (Phase 3a) 미결
    8 전면적 접근성 지원 스크린 리더, TalkBack 등 접근성 전면 지원 여부. Phase 3b 이후 별도 검토 미결
    9 다국어(i18n) 영어 등 다국어 지원 여부. 구조적 준비(리소스 분리)는 Phase 1에서 고려하되, 실제 번역 적용은 별도 논의 미결
    10 분석 및 로깅 분류 품질 관측 지표(5.9)를 Phase 2a부터 수집하기 위한 이벤트 로깅 체계를 사전 설계해야 함. 로깅 스키마 및 분석 파이프라인은 Phase 2a 착수 전 확정 대응 중

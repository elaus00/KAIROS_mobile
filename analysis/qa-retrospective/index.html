    Flit. QA 회고 리포트¶

    기간: 2025-01-23 ~ 2025-02-14 (23일) 세션: 169개 / 메시지: 660개 / QA 항목: 51건 데이터: qa-retrospective.csv (구조화 데이터)

    1. 요약¶

    23일간의 개발 과정에서 사용자가 지적한 QA 이슈 51건을 분석했다. 순수 구현 요청, 도구 질문, 문서 작업, Claude Code 자체 이슈는 제외하고, "잘못됐다", "안 된다", "고쳐라", "수정해" 등 문제 지적만 추출했다.

    3. 반복 패턴 Top 5¶

    같은 이슈가 여러 번 지적된 패턴을 분석했다. 반복 지적은 첫 수정이 불완전하거나, 근본 원인을 해결하지 않았을 때 발생한다.

    3.1 캘린더 관련 — 8회 반복 (최다)¶

    세부 이슈 반복 횟수 관련 항목
    패딩 불일치 3회 #1, #3, #10
    스와이프 미작동 2회 #24, #25
    헤더/주차 표시 2회 #3, #4
    상단 고정 스크롤 2회 #3, #27

    근본 원인: 캘린더는 주간/월간 뷰 전환, 스와이프 제스처, 날짜 선택 등 복합 인터랙션이 결합된 UI. 초기 설계 시 이 복잡도를 과소평가하여 부분 수정이 반복됨.

    교훈: 복잡한 인터랙션 UI는 구현 전 상태 다이어그램과 제스처 매핑을 정의하고, 한 번에 완성도 높게 구현해야 한다.

    3.2 네비게이션 바 — 5회 반복¶

    세부 이슈 관련 항목

    아이콘 크기/영역 #7
    폭 너무 큼 #14
    스크린샷 미표시 #40

    근본 원인: Phase 1에서 네비게이션 UI 셸만 배치하고 라우팅 미연결. 이후 점진적 수정마다 새로운 문제 발견.

    5. 근본 원인 분류¶

    51건의 근본 원인을 역추적하면 5가지 구조적 원인으로 수렴한다:

    5.1 디자인 가이드 부재 (02-08 이전) → 17건 유발¶

    02-08에 디자인 가이드가 생성되기 전까지 Material Design 기본값을 그대로 사용. 패딩, 타이포, 색상, 애니메이션 모두 사후 수정.

    • 영향 항목: #1~#14, #49
    • 이후 개선: 디자인 가이드 v2.0 수립 + CLAUDE.md 필수 참조 규칙 추가

    5.2 복합 UI의 점진적 구현 → 8건 유발¶

    캘린더처럼 상태가 많은 UI를 점진적으로 확장하면 매번 새로운 엣지 케이스 발견.

    • 영향 항목: #1, #3, #10, #23~#25, #27
    • 이후 개선: 상태 다이어그램 기반 설계 도입 필요 (미완)

    5.3 전역 기능의 부분 적용 → 4건 유발¶

    글씨 크기, 테마 색상 등 전역으로 작용해야 하는 기능이 일부 화면에만 적용.

    • 영향 항목: #12, #29, #30
    • 이후 개선: CompositionLocal 전파 + Theme 토큰 강제

    5.4 리브랜딩 수동 치환 → 4건 유발¶

    KAIROS → Flit 리브랜딩 시 전역 검색 대신 파일별 수동 치환으로 누락 발생.

    • 영향 항목: #41~#44
    • 이후 개선: 체크리스트 기반 전수 검사 + grep 검증

    5.5 Phase 1 셸 구현 → 5건 유발¶

    네비게이션 등 UI 셸만 배치하고 기능 연결을 다음 Phase로 미룬 결과, 사용자가 "눌러도 안 됨"을 경험.

    • 영향 항목: #18~#20
    • 이후 개선: 미구현 화면은 명시적 "준비 중" 표시 또는 네비에서 비활성화

    6. 핵심 교훈 5가지¶

    교훈 1: 디자인 시스템은 첫 번째 커밋 전에 수립하라¶

    디자인 가이드 없이 시작한 UI는 100% 재작업된다. 가이드 수립 전(01-23~02-08) QA 17건 vs 수립 후(02-08~02-14) UI QA 8건.

    교훈 2: 복잡한 인터랙션 UI는 한 번에 완성하라¶

    캘린더 8회 반복 지적의 원인 = 점진적 확장. 구현 전에 모든 상태 × 모든 제스처 매트릭스를 정의해야 한다.

    교훈 3: 전역 기능은 전역으로 검증하라¶

    글씨 크기 설정이 "일부 화면에서만" 적용되면 사용자는 "안 됨"으로 인식. 전역 기능은 모든 화면에서 테스트해야 한다.

    교훈 4: 대규모 치환은 자동화 + 검증이 필수다¶

    리브랜딩 4건의 잔재 = grep으로 30초면 발견 가능했던 이슈. grep -r "KAIROS" --include="*.kt" --include="*.xml" 한 줄이면 끝.

    교훈 5: 미구현 기능은 비활성화하거나 명시하라¶

    "눌러도 안 됨"은 버그와 구분할 수 없다. 아직 구현하지 않은 진입점은 회색 처리 + "준비 중" 라벨 또는 네비에서 제거.

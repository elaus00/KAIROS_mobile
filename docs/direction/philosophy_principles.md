## 1. 핵심 철학: Just Capture

### 1.1 철학의 본질

Kairos의 경쟁자는 다른 메모앱이 아니다. **"나중에 적어야지" 하다가 잊어버리는 것**, 그리고 **"또 정리 안 할 텐데" 하면서 기록을 포기하는 것**이다.

Kairos는 기록과 정리 사이의 연결고리를 끊는다. 사용자는 기록만 하면 된다. 정리는 AI가 한다. Kairos의 목표는 더 나은 기록 도구가 아니라, **기록을 포기하지 않게 만드는 시스템**이다.

### 1.2 세 가지 원칙

| 원칙 | 설명 | 의미 |
| --- | --- | --- |
| **Just Capture** | 앱 실행 → 즉시 입력. 어디에 적을지 고민할 필요 없이, 그냥 적으면 된다 | 기록의 마찰을 제거한다 |
| **Auto Organize** | 분류·태깅·목적지 결정은 AI가 처리한다. 정리되지 않은 메모가 쌓이지 않는다 | 정리 부담을 사용자에게서 가져간다 |
| **Straight to Action** | 일정은 캘린더로, 할 일은 투두로 자동 연결된다. 기록에서 행동까지의 거리를 없앤다 | 기록의 가치를 즉시 실현한다 |

### 1.3 철학이 해결하는 문제

| 문제 | 원칙 | 해결 방식 |
| --- | --- | --- |
| 분류 판단의 피로 | Just Capture | 사용자에게 분류를 요구하지 않는다 |
| 정리 부채의 누적 | Auto Organize | AI가 자동으로 분류하고 그룹화한다 |
| 기존 도구의 다단계 입력 | Just Capture | 앱 진입 시 즉시 입력 가능 |
| 대화 속 일정 누락 | Straight to Action | 공유 인텐트 → 자동 캘린더 등록 |
| 투두앱 방치 | Auto Organize + Straight to Action | 생성·분류·완료 흐름을 자동화 |
| 기록 가치 미회수 | Straight to Action | 기록이 행동(완료, 일정 등록)으로 이어진다 |

## 2. 설계 판단 기준

기능을 추가하거나 UI를 변경할 때 아래 질문에 하나라도 Yes이면 재고한다.

| 기준 | 질문 | 위반 시 결과 |
| --- | --- | --- |
| **Friction** | 이것이 기록까지의 단계를 늘리는가? | 사용자가 기록을 포기한다 |
| **Choice** | 사용자에게 선택을 강요하는가? | 판단 에너지를 소모하게 한다 |
| **Burden** | 정리 부담을 사용자에게 전가하는가? | 정리 부채가 누적된다 |
| **Flow** | 핵심 루프(기록 → 정리 → 실행)를 방해하는가? | 제품의 가치가 훼손된다 |
| **Necessity** | 이것 없이도 앱이 동작하는가? | 불필요한 복잡성이 추가된다 |
| **Quality** | AI 분류 품질을 비용 절감이나 편의를 위해 타협하는가? | 서비스 신뢰가 훼손된다 |

### 2.1 판단 기준 적용 사례

**사례 1: 캡처 시 유형 선택 요구**

| 기준 | 위반 여부 | 판단 |
| --- | --- | --- |
| Friction | Yes | 기록까지 한 단계가 추가됨 |
| Choice | Yes | 사용자가 유형을 결정해야 함 |

→ **기각.** AI 자동 분류로 대체한다.

**사례 2: 삭제 전 확인 다이얼로그**

| 기준 | 위반 여부 | 판단 |
| --- | --- | --- |
| Friction | Yes | 삭제에 한 단계가 추가됨 |
| Choice | Yes | "정말 삭제하시겠습니까?" 선택 강요 |
| Flow | Yes | 빠른 정리 흐름을 방해 |

→ **기각.** Snackbar 스타일 실행 취소로 대체한다.

**사례 3: 노트 보기 방식 3가지 제공**

| 기준 | 위반 여부 | 판단 |
| --- | --- | --- |
| Choice | 부분적 | 선택지가 많아짐 |
| Necessity | Yes | 하나의 기본 보기로도 동작함 |

→ **보류.** 사용자 피드백 기반으로 추후 검토한다.

**사례 4: AI 분류 사후 수정**

| 기준 | 위반 여부 | 판단 |
| --- | --- | --- |
| Choice | 아니오 | 강요가 아닌 옵션 |
| Burden | 아니오 | AI 오류 복구 수단이지 정리 부담 전가가 아님 |

→ **수용.** 에러 복구와 개인화를 위해 필요하다.

**사례 5: 분류 커스터마이징 (프리셋, 사용자 지시)**

| 기준 | 위반 여부 | 판단 |
| --- | --- | --- |
| Burden | 부분적 | 사용자가 규칙을 설정해야 함 |
| Necessity | Yes | 범용 분류만으로도 기본 동작함 |

→ **조건부 수용.** 고급 사용자 대상 구독 기능으로 위치시킨다. 기본 경험에 영향을 주지 않는다.

**사례 6: 일정 자동 추가 시 무조건 자동 등록**

| 기준 | 위반 여부 | 판단 |
| --- | --- | --- |
| Quality | Yes | 오분류 시 실제 약속 누락/더블부킹으로 이어짐 |
| Flow | 부분적 | 자동화를 위해 안전성을 포기함 |

→ **기각.** 신뢰도 기반 조건부 처리(자동 추가 vs 제안)로 대체한다. 기본값은 "제안" 모드.

**사례 7: 분류 변경 후 파생 객체를 사용자가 직접 정리**

| 기준 | 위반 여부 | 판단 |
| --- | --- | --- |
| Burden | Yes | 파생 객체 정리를 사용자에게 전가 |
| Flow | Yes | 분류 수정 → 캘린더 삭제 → 투두 확인의 다단계 작업 |

→ **기각.** 분류 변경 시 파생 객체는 자동으로 삭제/전환된다.

**사례 8: AI 모델 비용 절감을 위해 경량 모델로 전환**

| 기준 | 위반 여부 | 판단 |
| --- | --- | --- |
| Quality | Yes | 분류 정확도 저하로 사용자 신뢰 훼손 |

→ **기각.** 분류 품질은 서비스의 핵심이다. 비용 절감이 필요한 경우 품질 이외의 방법(캐싱, 배치 처리 등)을 우선 검토한다.

### 2.2 판단 기준 적용 체크리스트

새 기능을 제안할 때 다음 질문에 답한다.

1. 이 기능이 없으면 사용자가 핵심 루프를 완료할 수 없는가?
2. 이 기능이 사용자에게 추가 결정을 요구하는가?
3. 이 기능이 기록까지의 단계를 늘리는가?
4. 이 기능의 부담이 사용자에게 전가되는가?
5. 더 단순한 대안이 존재하는가?
6. 이 결정이 AI 분류 품질에 영향을 주는가?

## 3. 핵심 설계 원칙

설계 판단 기준을 구체적인 제품 영역에 적용한 원칙.

### 3.1 분류 신뢰 원칙

AI 분류는 서비스 전체의 신뢰를 결정하는 핵심 요소다.

| 원칙 | 설명 |
| --- | --- |
| 품질 최우선 | AI 모델 선정 시 분류 품질이 최우선이며, 비용 절감을 위해 품질을 타협하지 않는다 |
| 오분류 비용 최소화 | 오분류는 피할 수 없으므로, 오분류의 **인지 비용**과 **수정 비용**을 최소화한다 |
| 수정 마찰 1-2탭 | 분류 수정은 1-2탭으로 완료 가능해야 한다 |
| 파생 객체 일관성 | 분류 변경 시 파생 객체(캘린더 이벤트, 투두 등)는 자동으로 삭제/전환된다 |
| 관측 우선 | 분류 품질은 출시 시점부터 관측 지표(수정률, 승인율 등)로 측정한다 |

### 3.2 Inbox 원칙

Inbox는 AI 미분류 항목의 임시 저장소이며, 그 자체가 정리 부채가 되어서는 안 된다.

| 원칙 | 설명 |
| --- | --- |
| 비전면 배치 | 메인 네비게이션에 전면 노출하지 않는다 |
| 비침습적 알림 | 잔류 항목이 있을 때 부드러운 nudging만 제공한다 |
| AI 자동 해소 | AI 주기적 재분류가 Inbox를 자동으로 비우는 것이 기본 동작이다 |
| 사용자 관리 최소화 | 사용자가 Inbox를 직접 관리해야 하는 빈도를 최소화한다 |

### 3.3 일정 안전 원칙

일정은 다른 분류 유형보다 오분류의 실수 비용이 크다 (더블부킹, 약속 누락).

| 원칙 | 설명 |
| --- | --- |
| 조건부 자동화 | 자동 추가는 신뢰도(High)인 경우에만. 그 외는 제안(1탭 확인)으로 처리 |
| 기본값은 안전 | 기본 설정은 "제안" 모드. 완전 자동은 사용자 opt-in |
| 실행 취소 항상 제공 | 자동 추가된 일정은 항상 실행 취소가 가능해야 한다 |
| Just Capture 유지 | 안전장치가 캡처 행위 자체에 마찰을 추가하지 않는다 |

### 3.4 원본 보존 원칙

AI가 캡처 내용을 가공(제목 생성, 요약 등)하더라도 원본은 항상 보존한다.

| 원칙 | 설명 |
| --- | --- |
| 원본 텍스트 보존 | AI 제목이 생성되더라도 캡처 원문은 항상 저장되며 상세 보기에서 확인 가능 |
| AI 가공 투명성 | AI가 생성한 제목/태그는 사용자가 수정할 수 있다 |
| 비파괴적 처리 | AI 처리가 사용자의 원본 데이터를 변경하거나 손실시키지 않는다 |

### 3.5 회수 가능성 원칙

기록의 가치는 나중에 찾아서 사용할 수 있을 때 실현된다.

| 원칙 | 설명 |
| --- | --- |
| 스캔 가능한 리스트 | 리스트에서 항목을 빠르게 식별할 수 있어야 한다 (AI 제목, 분류 칩, 태그) |
| 검색 인프라 | 제목, 원문, 태그, 엔티티 모두 검색 대상이다 |
| 메타데이터 품질 | AI 제목과 태그의 품질이 회수 성공률을 결정한다 |

## 4. 디자인 가이드라인

### 4.1 입력 UX

| 원칙 | 적용 |
| --- | --- |
| 즉시 접근 | 앱 진입 시 입력창이 첫 화면에 위치 |
| 최소 입력 | 텍스트만 입력하면 완료. 부가 정보는 AI가 추론 |
| 입력 보호 | 앱 이탈 시 자동 저장, 재진입 시 복원 |
| 빠른 피드백 | 저장 완료 즉시 시각적 확인 |

### 4.2 분류 UX

| 원칙 | 적용 |
| --- | --- |
| 자동 우선 | AI가 먼저 분류하고, 사용자는 필요 시 수정 |
| 시각적 표시 | 분류 결과를 칩으로 명확히 표시 |
| 낮은 수정 마찰 | 칩 탭 → 유형 변경 시트 → 선택으로 1-2탭 완료 |
| 안전한 폴백 | 미분류는 Inbox에 저장, 손실 없음 |
| 파생 일관성 | 분류 변경 시 파생 객체 자동 처리 |

### 4.3 리스트 UX

| 원칙 | 적용 |
| --- | --- |
| AI 제목 우선 | 리스트에는 AI 생성 제목을 표시 (원문이 짧으면 원문 그대로) |
| 프리뷰 제공 | 제목과 원문이 다른 경우 원문 첫 줄을 프리뷰로 표시 |
| 분류 칩 항상 표시 | 어떤 유형인지 즉시 식별 가능 |
| 시간 컨텍스트 | 캡처 시점 또는 마감 시간을 표시하여 시간 맥락 제공 |

### 4.4 삭제 UX

| 원칙 | 적용 |
| --- | --- |
| 확인 다이얼로그 금지 | 삭제 전 "정말 삭제하시겠습니까?"를 묻지 않는다 |
| 실행 취소 제공 | 삭제 후 Snackbar로 5초간 "실행 취소" 제공 |
| 비파괴적 삭제 | 실행 취소 기간 내 데이터 유지 |

### 4.5 알림 UX

| 원칙 | 적용 |
| --- | --- |
| 정보 제공 | 자동 처리 결과를 사후에 알린다 (사전 확인 요청 아님) |
| 비침습적 | 앱 내 알림은 콘텐츠를 가리지 않는다 |
| 행동 연결 | 푸시 알림 탭 시 해당 항목으로 바로 이동 |
| 일정 제안은 1탭 | 일정 제안 알림에서 1탭으로 승인/거부 가능 |

### 4.6 오프라인 UX

| 원칙 | 적용 |
| --- | --- |
| 캡처 항상 성공 | 네트워크 상태와 무관하게 로컬 저장 |
| 상태 명시 | 오프라인 상태를 앱 내에서 명확히 표시 |
| 자동 복구 | 네트워크 복구 시 AI 분류 자동 재시도 |

## 5. UX 라이팅 원칙

### 5.1 톤

| 원칙 | 설명 | 예시 |
| --- | --- | --- |
| 간결함 | 불필요한 단어를 제거한다 | ❌ "성공적으로 저장되었습니다" → ✅ "저장됨" |
| 행동 중심 | 상태보다 행동을 안내한다 | ❌ "오프라인 상태입니다" → ✅ "오프라인 — 연결되면 자동 동기화" |
| 긍정적 | 오류도 긍정적으로 표현한다 | ❌ "분류 실패" → ✅ "Inbox에 저장됨 — 연결 후 자동 분류" |

### 5.2 용어 통일

| 개념 | 용어 | 피해야 할 표현 |
| --- | --- | --- |
| 기록 행위 | 캡처 | 메모, 입력, 작성 |
| 임시 저장소 | Inbox | 미분류함, 대기함 |
| 아이디어 분류 | ideas | 아이디어함, 발상함 |
| AI 재분류 | 재정리 | 다시 정리, 리밸런싱 |
| 삭제 복구 | 실행 취소 | 되돌리기, 복원 |
| 일정 확인 요청 | 제안 | 추천, 알림 |
| 분류 유형 표시 | 칩 | 태그, 뱃지, 라벨 |

### 5.3 에러 메시지

| 상황 | 메시지 | 원칙 |
| --- | --- | --- |
| 네트워크 오류 | "오프라인 — Inbox에 저장됨" | 문제 + 결과를 함께 |
| AI 타임아웃 | "분류 중 — 잠시 후 다시 시도합니다" | 진행 상태 표시 |
| 미지원 파일 | "텍스트와 이미지만 지원됩니다" | 대안 없이 간결하게 |
| 캘린더 연동 실패 | "캘린더 추가 실패 — 다시 시도합니다" | 자동 재시도 안내 |
